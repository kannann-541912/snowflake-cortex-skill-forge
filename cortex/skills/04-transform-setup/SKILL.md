# Safety
- Source queries read from `SNOWFLAKE_SAMPLE_DATA.TPCH_SF10` — read-only.
- All generated models (Dynamic Tables, dbt models) write to `SANDBOX.TPCH`.

# TPCH Transform Patterns
Because the source is a live structured database (not a stage), the transform strategy
uses **CTAS / Dynamic Tables / dbt SELECT FROM source** directly — no COPY INTO needed.

Key TPCH column type corrections to always apply:
| Source Column Pattern | Cast To | Reason |
|---|---|---|
| O_TOTALPRICE, L_EXTENDEDPRICE, etc. | NUMBER(15,2) | Prices stored as FLOAT in source |
| O_ORDERDATE, L_SHIPDATE, L_COMMITDATE, L_RECEIPTDATE | DATE | Already DATE in TPCH |
| C_ACCTBAL, PS_SUPPLYCOST | NUMBER(15,2) | Financial values |
| *_COMMENT columns | VARCHAR(256) | Free text, keep but deprioritise |

# When to Use
- User has completed `$de-schema-setup` and objects exist in Snowflake
- User says "set up transforms", "map the columns", "build the transformation logic", "create dbt models"
- Defining how raw source data becomes clean target data

# What This Skill Provides
Replaces bespoke, fragile per-source scripts with reusable, declarative transform mappings.
Produces a transform_mappings.yml + dbt model stubs + Snowflake Dynamic Table DDL.

# Instructions

## Step 1 — Load context
Read `schema_design.md` and `schema_setup.sql`.
Inspect actual source and target columns via:
```sql
SHOW COLUMNS IN TABLE {source_table};
SHOW COLUMNS IN TABLE {target_table};
```

## Step 2 — Build transform mapping document
For each source → target column pair, document the transform rule.
Write `transform_mappings.yml`:

```yaml
# transform_mappings.yml
# Generated by $de-transform-setup at {timestamp}

source: "{source_db}.{source_schema}.{source_table}"
target: "{target_db}.{target_schema}.{target_table}"

column_mappings:
  - source_col: ORDER_ID
    target_col: ORDER_ID
    transform: "direct"                          # no change
    cast: "NUMBER(38,0)"
    nullable: false

  - source_col: ORDER_DATE
    target_col: ORDER_DATE
    transform: "cast"
    expression: "TRY_TO_TIMESTAMP_LTZ(ORDER_DATE)"
    cast: "TIMESTAMP_LTZ"
    nullable: false
    on_null: "CURRENT_TIMESTAMP()"              # fallback if null

  - source_col: AMOUNT
    target_col: ORDER_AMOUNT_USD
    transform: "expression"
    expression: "ROUND(AMOUNT::FLOAT, 2)"
    cast: "NUMBER(18,2)"
    nullable: false

  - source_col: STATUS
    target_col: ORDER_STATUS
    transform: "lookup"
    expression: |
      CASE STATUS
        WHEN 'C' THEN 'COMPLETED'
        WHEN 'P' THEN 'PENDING'
        WHEN 'X' THEN 'CANCELLED'
        ELSE 'UNKNOWN'
      END
    cast: "VARCHAR(50)"
    nullable: false

  - source_col: CUSTOMER_EMAIL
    target_col: CUSTOMER_EMAIL
    transform: "direct"
    cast: "VARCHAR(256)"
    pii: true
    masking_policy: "MASK_EMAIL"

  # Derived columns (no source column)
  - source_col: null
    target_col: _LOADED_AT
    transform: "derived"
    expression: "CURRENT_TIMESTAMP()"

  - source_col: null
    target_col: _SOURCE_SYSTEM
    expression: "'{source_name}'"
```

## Step 3 — Generate dbt staging model
Write `.cortex/dbt/models/staging/stg_{source_name}.sql`:

```sql
-- stg_{source_name}.sql
-- dbt model: staging layer
-- Materialisation: view
-- Generated by $de-transform-setup

{{{{ config(materialized='view') }}}}

WITH source AS (
    SELECT * FROM {{{{ source('{source_schema}', '{source_table}') }}}}
),

renamed AS (
    SELECT
        -- IDs
        ORDER_ID::NUMBER(38,0)                     AS order_id,

        -- Dates
        TRY_TO_TIMESTAMP_LTZ(ORDER_DATE)           AS order_date,

        -- Measures
        ROUND(AMOUNT::FLOAT, 2)                    AS order_amount_usd,

        -- Lookups
        CASE STATUS
            WHEN 'C' THEN 'COMPLETED'
            WHEN 'P' THEN 'PENDING'
            WHEN 'X' THEN 'CANCELLED'
            ELSE 'UNKNOWN'
        END                                        AS order_status,

        -- PII (masked at table level via policy)
        CUSTOMER_EMAIL                             AS customer_email,

        -- Audit
        CURRENT_TIMESTAMP()                        AS _loaded_at,
        '{source_name}'                            AS _source_system

    FROM source
)

SELECT * FROM renamed
```

## Step 4 — Generate Snowflake Dynamic Table DDL
For large tables or near-real-time needs, also generate a Dynamic Table alternative:

```sql
-- dynamic_table_{target_table}.sql
CREATE OR REPLACE DYNAMIC TABLE {target_db}.{target_schema}.{target_table}
  TARGET_LAG = '1 hour'        -- adjust: '1 minute' to '7 days'
  WAREHOUSE = ANALYTICS_WH
  REFRESH_MODE = AUTO
AS
SELECT
    ORDER_ID::NUMBER(38,0)               AS order_id,
    TRY_TO_TIMESTAMP_LTZ(ORDER_DATE)     AS order_date,
    ROUND(AMOUNT::FLOAT, 2)              AS order_amount_usd,
    CASE STATUS
        WHEN 'C' THEN 'COMPLETED'
        WHEN 'P' THEN 'PENDING'
        WHEN 'X' THEN 'CANCELLED'
        ELSE 'UNKNOWN'
    END                                  AS order_status,
    CUSTOMER_EMAIL,
    CURRENT_TIMESTAMP()                  AS _loaded_at,
    '{source_name}'                      AS _source_system
FROM {source_db}.{source_schema}.{source_table};
```

## Step 5 — Test transforms against sample data
Run transforms against 100 rows to validate:
```sql
SELECT
    ORDER_ID::NUMBER(38,0)               AS order_id,
    TRY_TO_TIMESTAMP_LTZ(ORDER_DATE)     AS order_date,
    -- ... full transform expression
FROM {source_table}
LIMIT 100;
```
Report: rows processed, null counts in NOT NULL target columns, any CAST failures.


## Best Practices
- Use `TRY_TO_*` cast functions — they return NULL on failure instead of erroring
- Map UNKNOWN/unexpected values explicitly in CASE statements — never let them silently become NULL
- Document every non-trivial expression in a comment
- Test transforms on sample data before full load

## Common Patterns

### Pattern 1: Date normalization
`TRY_TO_TIMESTAMP_LTZ(date_col, 'MM/DD/YYYY')` for non-ISO date strings

### Pattern 2: Numeric cleaning
`TRY_TO_NUMBER(REGEXP_REPLACE(amount_col, '[^0-9.]', ''))` for dirty currency strings
